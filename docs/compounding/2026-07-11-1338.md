### [C-PORT1] Verify the Codex skill discovery directory against a real Codex install
- **Severity:** medium
- **Effort:** low
- **Pickup:** operator-action
- **Ready:** no
- **Files:** `scripts/install-skills`, `README.md`
- **Status:** OPEN

**Background:** the installer defaults `CODEX_SKILLS_DIR` to `~/.agents/skills` (the cross-agent
convention). Codex documentation has historically used `$CODEX_HOME/skills` (`~/.codex/skills`),
and not every Codex version reads the shared directory. If the default is wrong, every Codex
install silently discovers zero skills — the exact silent no-op this repository's own hygiene
rules warn about. This cannot be verified from a session without a Codex install.

**Goal:** the installer's default Codex target matches what the operator's installed Codex
actually discovers, verified once on a real machine.

**Acceptance criteria:**
- [ ] On a machine with Codex installed, run `scripts/install-skills --target codex` and confirm
      Codex lists the installed skills (e.g. `$catch-up` resolves)
- [ ] If discovery fails, either change the default directory or link both candidate directories,
      and record which Codex version was verified in README's install section
- [ ] `sync-ai-tools` step 4 gains a line: parity means each runtime *lists* the skill, not merely
      that a symlink resolves

### [C-PORT2] Define and execute the retirement criterion for legacy commands/ wrappers
- **Severity:** low
- **Effort:** low
- **Pickup:** operator-action
- **Ready:** no
- **Files:** `commands/*.md`, `scripts/install-skills`, `AGENTS.md`, `README.md`
- **Status:** OPEN

**Background:** AGENTS.md preserves top-level `commands/` wrappers "until their replacements are
installed and verified in both Claude Code and Codex", but nothing records what verification means
or when it happened, so the wrappers can live forever by default. They also hard-depend on the
`~/.ai-tools` anchor existing: in a cloud or fresh-clone session the wrapper points at a missing
path, while native skill discovery works without it.

**Goal:** legacy wrappers have a concrete, recorded exit condition and are removed when it is met.

**Acceptance criteria:**
- [ ] Operator confirms native skill invocation works in both daily-driver runtimes (Claude Code
      slash-name and Codex `$name`) for at least the lifecycle skills
- [ ] After confirmation: delete top-level `commands/*.md` wrappers, drop the installer's legacy
      command-link step, and update AGENTS.md/README (compounding-templates/ is NOT in scope —
      it migrates with the template pack)
- [ ] `scripts/install-skills` prune removes the now-dangling `~/.claude/commands` links on the
      next run (already implemented; verify it fires)

### [C-PORT3] Decide whether portable skills get a provider-frontmatter escape hatch
- **Severity:** low
- **Effort:** low
- **Pickup:** operator-action
- **Ready:** no
- **Files:** `docs/decisions/0003-portable-cores-with-provider-capability-adapters.md`, `scripts/validate-skills.mjs`, `AGENTS.md`
- **Status:** DONE (PR #22)

**Background:** the portable frontmatter rule (only `name` + `description`) means a skill can never
use Claude Code's native frontmatter capabilities (e.g. tool allowlists or invocation controls),
and there is no adapter escape hatch for frontmatter the way there is for scheduling and
connectors. This is the one place the setup is lowest-common-denominator despite decision 0003's
"portability does not mean lowest-common-denominator" principle. It may be the right tradeoff —
but it is currently an accident of the validator, not a recorded decision.

**Goal:** decision 0003 either affirms the two-key frontmatter rule explicitly or defines the
mechanism for provider-specific skill metadata (e.g. an `agents/claude.yaml` sibling of
`agents/openai.yaml`), and the validator enforces whichever is chosen.

**Acceptance criteria:**
- [x] Decision 0003 gains a section covering provider frontmatter/skill-metadata, with the chosen
      direction and rationale
- [x] The existing two-key rule is affirmed: no validator or AGENTS.md code-path change; decision
      text now makes the boundary and documented-sidecar exception explicit

**Proposed approach:** mirror the existing Codex pattern — provider skill metadata lives under
`agents/<provider>.yaml`, never in portable frontmatter — so both runtimes get native features
through the same shaped adapter.

**Decision (2026-07-14):** do not create a generic shaped adapter that runtimes may ignore. Keep
portable frontmatter to `name` and `description`; use a provider sidecar only when that provider
documents and consumes it (currently Codex's `agents/openai.yaml`).
